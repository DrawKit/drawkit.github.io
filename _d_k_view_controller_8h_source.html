123 /*
124 
125 DKViewController is a basic controller class that sits between a DKDrawingView and the DKDrawing itself, which implements the data model.
126 
127 Its job is broadly divided into two areas, input and output.
128 
129 When part of a drawing needs to be redisplayed in the view, the drawing will pass the area needing update to the controller, which will
130 set that area for redisplay in the view, if appropriate. The view redisplays the content accordingly (it may call DKDrawing's drawRect:inView: method).
131 Other subclasses of this might present the drawing differently - for example a layers palette could display the layers as a list in a tableview.
132 
133 Each view of the drawing has one controller, so the drawing has a to-many relationship with its controllers, but each controller has a
134 to-one relationship with the view.
135 
136 An important function of the controller is to receive user input from the view and direct it to the active layer in an appropriate way. This
137 includes handling the "tool" that a user might select in an interface and applying it to the drawing. See DKToolController (a subclass of this).
138 This also implements autoscrolling around the mouse down/up calls which by and large "just work". However if you override these methods you should
139 call super to keep autoscrolling operative.
140 
141 Ownership: drawings own the controllers which reference the view. Views keep a reference to their controllers. When a view is dealloc'd, its
142 controller is removed from the drawing. The controller has weak references to both its view and the drawing - this permits a view to own a drawing
143 without a retain cycle being introduced - whichever of the drawing or the view gets dealloc'd first, the view controller is also dealloc'd. A view can
144 own a drawing in the special circumstance of a view creating the drawing automatically if none has been set up prior to the first call to -drawRect:
145 
146 Flow of control: initially all messages that cannot be directly handled by DKDrawingView are forwarded to its controller. The controller can
147 handle the message or pass it on to the active layer. This is the default behaviour - typically layer subclasses handle most of their own
148 action messages and some handle their own mouse input. For most object layers, where a "tool" can be applied, the controller works with the tool
149 to implement the desired behaviour within the target layer. The view and the controller both use invocation forwarding to push messages down
150 into the DK system via the controller, the active layer, any selection within it, and finally the target object(s) there.
151 
152 A subclass of this can also implement drawRect: if it needs to, and can thus draw into its view. This is called after all other drawing has been
153 completed except for page breaks. Tool controllers for example can draw selection rects, etc.
154 
155 */
