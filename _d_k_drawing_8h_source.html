429 /*
430 
431 A DKDrawing is the model data for the drawing system. Usually a document will own one of these. A drawing consists of one or more DKLayers,
432 each of which contains any number of drawable objects, or implements some special feature such as a grid or guides, etc.
433 
434 A drawing can have multiple views, though typically it will have only one. Each view is managed by a single view controller, either an instance
435 or subclass of DKViewController. Drawing updates refersh all views via their controllers, and input from the views is directed to the current
436 active layer through the controller. The drawing owns the controllers, but the views are owned as normal by their respective superviews. The controller
437 provides only weak references to both drawing and view to prevent potential retain cycles when a view owns a drawing for the automatic backend scenario.
438 
439 The drawing and the attached views must all have the same bounds size (though the views are free to have any desired frame). Setting the
440 drawing size will adjust the views' bounds automatically.
441 
442 The active layer will receive mouse events from any of the attached views via its controller. (Because the user can't mouse in more than one view
443 at a time, there is no contention here.) The commands will go to whichever view is the current responder and be passed on appropriately.
444 
445 Drawings can be saved simply by archiving them, thus all parts of the drawing need to adopt the NSCoding protocol.
446 
447 */
