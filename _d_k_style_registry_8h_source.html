136 /*
137 
138 The style registry is a singleton category manager instance that consolidates styles from a variety of sources into a single app-wide "database"
139 of styles, organised into categories.
140 
141 Styles can come from these sources:
142 
143 1. The application defaults, if the app is launched with no library preferences available (i.e. first run).
144 
145 2. The styles library, which is the complete registry saved to disk (user prefs) at quit time.
146 
147 3. A document, when it is opened.
148 
149 4. A separate file containing just styles.
150 
151 5. A new style being created and registered by the user as the app is used.
152 
153 -------------------------------------------------------------------------------------------------------------------------------------------------
154 The point of the registry is twofold:
155 
156 A. It permits the construction of a UI for accessing pre-built styles and applying them to objects in a drawing. By organising styles into categories,
157 potentially large number of styles may be managed effectively.
158 
159 B. It tracks styles across any number of documents as they are created. For example if a document uses a particular registered style, when that
160 document is reopened at a later date, the style can be recognised and linked to the same style in the registry. If the style has changed in the
161 meantime so that there is a difference between the saved style and the currently registered version, the user can be offered the option to
162 update the style in the document to match the registry, update the registry to match the document, or to re-register the style as a new version.
163 
164 Note the registry is not mandatory - an app using DK can use styles without registering them if it wishes. The advantage of the registry is that it
165 permits styles to persist and be tracked across multiple documents, saving the user from having to redefine styles for every new graphic.
166 
167 -------------------------------------------------------------------------------------------------------------------------------------------------
168 In order for the registry to uniquely and positively identify a style, its unique ID is used as its key. The unique ID is assigned when the style
169 first comes into existence and cannot be changed. It is a string representation of a UUID so is guaranteed unique.
170 
171 UUID's are not very user friendly and should never be exposed by an application's user interface. Thus a style also has an ordinary descriptive
172 name which can be displayed in the UI. Such names are not guaranteed to be unique however, as the user is free to enter whatever name they wish.
173 When a style is first registered the name may be changed to avoid a collision with an already registered style having the same name - this is
174 done by appending 1, 2, 3 etc until the name doesn't collide. However this is just done to disambiguate the style for the user - it is not intended
175 to guarantee uniqueness so that the name can be used as a key to the object.
176 
177 A user interface will want to use the ordinary names, but internally must be set up to use the unique ID or the object itself to avoid any confusion
178 as to which style is actually being referred to. For example a menu of available styles could use the UUID or the object as the represented object
179 of the menu item. DO NOT USE THE ORDINARY NAME AS A KEY.
180 
181 -------------------------------------------------------------------------------------------------------------------------------------------------
182 
183 Locking. Styles in the registry are usually locked. This is to prevent accidental alteration of a style that may be being used across many
184 documents. When a style is added to the registry, it should be unlocked (because the name might need to be automatically changed) but the
185 registry will lock the style upon a successful operation.
186 
187 Styles in a document. Styles used in a document may or may not be registered. A user may never register a style, but still have many styles
188 defined in a document. Such styles work normally and can be copied and pasted and shared between objects if set to do so - use of styles is not
189 dependent in any way upon the registry. When the document is saved, a flag as to whether the style was registered or not is saved with it. When
190 the document is opened again later, such flagged styles will be optionally reconnected to the style registry so that the actual style used is
191 always the registered style, so if the style is updated, existing documents are offered the opportunity to use the updated version.
192 
193 As well as any user-defined categories, documents may wish to create a temporary category using the document's name which gives the user a
194 way to quickly discover the complete set of registered styles used in a document. The category should be removed when the document is closed.
195 DKDrawDocument implements this behaviour by default, so if your document class is based on it, your app can get this feature for free.
196 
197 -------------------------------------------------------------------------------------------------------------------------------------------------
198 
199 Registering a style. The style registry performs the following steps:
200 
201 1. It checks that the style is not already registered. If it is, it does no more.
202 
203 2. It checks that the style is unlocked. If not, an exception is thrown.
204 
205 3. It resolves the style's name so that there are no collisions with any existing style's name, and changes the style's name if needed.
206 
207 4. It creates any new categories as requested (depends on the particular method used to register the style). If no specific categories are
208  requested, the style is added to the default category.
209 
210 5. It adds the style to the registry using its unique ID as the key.
211 
212 6. It locks the style.
213 
214 -------------------------------------------------------------------------------------------------------------------------------------------------
215 
216 Cut/Paste: cut and paste of styles works independently of the registry, including dealing with shared styles. See DKStyle for more info.
217 
218 
219 */
