34 // when a tool performs undoable actions, it doesn't mean it necessarily WILL perform an undoable action. Since complex tasks are usually
35 // grouped, there needs to be a way to start a group at the right time, if and only if there WILL be something undoable. Unfortunately this
36 // is required because NSUndoManager has a bug where opening and closing a group but doing nothing in between records a bogus undo task.
37 
38 // Thus a tool can signal to its delegate that the operation it is about to perform will create an undo task, and so the delegate can

48 /*
49 
50 The drawing tool protocol must be implemented by all tools that can be used to operate on a drawing. Getting tools right is tricky,
51 because of all the different cases that need to be considered, undo tasks, and so forth. Thus the following rules must be followed:
52 
53 1. On mouseDown, a tool needs to decide what it is going to do, and return the partcode of the hit part for the object under consideration. At
54 this point however, it should NOT perform the actual action of the tool.
55 
56 2. The partcode returned in 1, if non-zero, will be passed back during a mouse drag.
57 
58 3. On mouse UP, the tool must carry out its actual action, returning YES if the action was carried out, NO if not. The correct return values
59 from mouse down and mouse up are essential to allow the correct management of undo tasks that arise during the tool's operation.
60 
61 4. Tools that do not affect the data content of a drawing (e.g. a zoom tool, which affects only the view) should return 0 and NO respectively.
62 
63 5. Tools that perform an action that can be considered undoable must implement +toolPerformsUndoableAction returning YES and also supply an
64 action name when requested.
65 
66 6. Tools must supply a cursor which is displayed during the mouse down/drag/up sequence and whenever the tool is set.
67 
68 */
