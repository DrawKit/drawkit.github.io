84 /*
85 
86 This object is a view controller that can apply one of a range of tools to the objects in the currently active drawing layer.
87 
88 ==== WHAT IS A TOOL? ====
89 
90 Users "see" tools often as a button in a palette of tools, and can choose which tool is operative by clicking the button. While your
91 application may certainly implement a user interface for selecting among tools in this way, DK's concept of a tool is more abstract.
92 
93 In DK, a tool is an object that takes basic mouse events that originate in a view and translates those events into meaningful operations
94 on the data model or other parts of DK. Thus a tool is essentially a translator of mouse events into specific behaviours. Different tools have
95 different behaviours, but all adopt the same basic DKDrawingTool protocol. Tools are part of the controller layer of the M-V-C
96 paradigm.
97 
98 Not all tools necessarily change the data content of the drawing. For example a user might pick a zoom tool from the same palette that
99 has other drawing tools such as rects or ovals. A zoom tool doesn't change the data content, it only changes the state of the view. The
100 tool protocol permits the controller to determine whether the data content was changed so it can help manage undo and so forth.
101 
102 Tools may optionally draw something in the view - if so, they are given the opportunity to do so after all other drawing, so tools draw
103 "on top" of any other content. Typically a tool might draw a selection rect or similar.
104 
105 Tools are responsible for applying their own behaviour to the target object(s), this controller merely calls the tool appropriately.
106 
107 ==== CHOOSING TOOLS ====
108 
109 This controller permits one tool at a time to be set. This can be applied globally for the whole application, on a per-document (drawing)
110 basis, or individually for the view. Which you use will depend on your needs and the sort of user interface that your application wants
111 to implement for tools. DK provides no UI and makes no assumptions about it - your UI is required to somehow pick a tool and set it.
112 
113 Tools can be stored in a registry (see DKDrawingTool) using a name. A UI may take advantage of this by using the name to look up the
114 tool and set it. As a convenience, the -selectDrawingToolByName: action method will use the -title property of <sender> as the name and
115 set the tool if one exists in the registry with this name - thus a palette of buttons for example can just set each button title to the
116 tool's name and target first responder with this action.
117 
118 */
