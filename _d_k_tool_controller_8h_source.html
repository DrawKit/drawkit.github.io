191 /*
192 
193 This object is a view controller that can apply one of a range of tools to the objects in the currently active drawing layer.
194 
195 ==== WHAT IS A TOOL? ====
196 
197 Users "see" tools often as a button in a palette of tools, and can choose which tool is operative by clicking the button. While your
198 application may certainly implement a user interface for selecting among tools in this way, DK's concept of a tool is more abstract.
199 
200 In DK, a tool is an object that takes basic mouse events that originate in a view and translates those events into meaningful operations
201 on the data model or other parts of DK. Thus a tool is essentially a translator of mouse events into specific behaviours. Different tools have
202 different behaviours, but all adopt the same basic DKDrawingTool protocol. Tools are part of the controller layer of the M-V-C
203 paradigm.
204 
205 Not all tools necessarily change the data content of the drawing. For example a user might pick a zoom tool from the same palette that
206 has other drawing tools such as rects or ovals. A zoom tool doesn't change the data content, it only changes the state of the view. The
207 tool protocol permits the controller to determine whether the data content was changed so it can help manage undo and so forth.
208 
209 Tools may optionally draw something in the view - if so, they are given the opportunity to do so after all other drawing, so tools draw
210 "on top" of any other content. Typically a tool might draw a selection rect or similar.
211 
212 Tools are responsible for applying their own behaviour to the target object(s), this controller merely calls the tool appropriately.
213 
214 ==== CHOOSING TOOLS ====
215 
216 This controller permits one tool at a time to be set. This can be applied globally for the whole application, on a per-document (drawing)
217 basis, or individually for the view. Which you use will depend on your needs and the sort of user interface that your application wants
218 to implement for tools. DK provides no UI and makes no assumptions about it - your UI is required to somehow pick a tool and set it.
219 
220 Tools can be stored in a registry (see DKDrawingTool) using a name. A UI may take advantage of this by using the name to look up the
221 tool and set it. As a convenience, the -selectDrawingToolByName: action method will use the -title property of <sender> as the name and
222 set the tool if one exists in the registry with this name - thus a palette of buttons for example can just set each button title to the
223 tool's name and target first responder with this action.
224 
225 */
